# Implementation Plan: Cloudflare R2 Project Export & Download

## Overview

Persist and serve user-created projects (files generated by the agent) using Cloudflare R2. Each session gets an isolated workspace; when a session completes, the workspace is zipped and uploaded to R2. Users can download their project via a pre-signed URL.

## Design Summary

| Concern | Approach |
|---------|----------|
| Isolation | Per-session workspace: `{baseWorkingDir}/{sessionId}` |
| Persistence | Zip workspace on completion → upload to R2 |
| Scale | R2 handles storage; no egress fees; pre-signed URLs offload bandwidth |
| Large projects | Stream zip creation; multipart upload for objects > 5MB |

---

## Architecture

```
┌─────────────┐     session_complete      ┌─────────────┐
│   Agent     │  ──────────────────────► │   Server    │
│             │  (terminal status)        │             │
│ /workspace/ │                           │ Export job  │
│  {sessionId}│                           │ 1. Zip dir  │
└─────────────┘     shared volume         │ 2. Upload   │
       │                    │             │ 3. Save     │
       │                    ▼             │    Project  │
       └────────────► /workspace ◄────────┘             │
                      (shared volume)                    │
                               │                         │
                               └─────────────────────────┘
                                        │
                                        ▼
                               ┌─────────────────┐
                               │ Cloudflare R2   │
                               │ projects/       │
                               │  {sessionId}.zip│
                               └─────────────────┘
```

## Implementation Steps

### Step 1: Prisma Schema – Add Project Model

**File:** `packages/server/prisma/schema.prisma`

Add new model and session relation:

```prisma
model Project {
  id         String   @id @default(uuid())
  sessionId String   @unique @map("session_id")
  session    Session  @relation(fields: [sessionId], references: [id], onDelete: Cascade)
  blobKey    String   @map("blob_key")
  sizeBytes  Int      @map("size_bytes")
  expiresAt DateTime? @map("expires_at")
  createdAt  DateTime @default(now()) @map("created_at")

  @@map("projects")
}

// Add to Session model:
  project Project?
```

Run migration: `npx prisma migrate dev --name add_project_export`

---

### Step 2: Per-Session Workspace in Agent

**Files:** `packages/agent/src/ws-client.ts`, `packages/agent/src/agent-loop.ts`

**Current:** Agent uses a single `workingDir` for all sessions (files overwrite across sessions).

**Change:** Use `{baseWorkingDir}/{sessionId}` as the effective workspace for each session.

1. In `ws-client.ts`, when handling `server:session:assign` and `server:session:continue`:
   - Compute `sessionWorkingDir = path.join(workingDir, message.sessionId)`
   - Ensure directory exists: `fs.mkdirSync(sessionWorkingDir, { recursive: true })`
   - Pass `sessionWorkingDir` to `runAgentLoop` instead of `workingDir`

2. No changes to `agent-loop.ts` or tools — they already receive the working dir as a parameter.

---

### Step 3: Shared Volume in Docker

**File:** `docker-compose.yml`

Add a named volume and mount it on both agent and server:

```yaml
services:
  server:
    # ... existing config ...
    volumes:
      - workspace:/workspace
    environment:
      # ... existing ...
      WORKSPACE_DIR: /workspace

  agent:
    # ... existing config ...
    volumes:
      - workspace:/workspace
    environment:
      # ... existing ...
      # Working dir stays /workspace (base); session subdirs created at runtime

volumes:
  pgdata:
  workspace:
```

**File:** `packages/agent/Dockerfile` – CMD stays `--working-dir /workspace`

### Step 4: R2 Client & Export Service

**File:** `packages/server/package.json`

Add dependencies:

```json
"@aws-sdk/client-s3": "^3.x",
"archiver": "^7.x"
```

**New file:** `packages/server/src/services/export-service.ts`

```typescript
// Key functions:
// - exportSessionWorkspace(sessionId: string): Promise<Project>
//   - Read /workspace/{sessionId} (from env WORKSPACE_DIR)
//   - Create zip stream with archiver (streaming, no full buffer)
//   - Upload to R2 via S3 client (multipart if > 5MB)
//   - Return Project record
// - getPresignedDownloadUrl(projectId: string, expiresInSeconds?: number): Promise<string>
```

R2 is S3-compatible. Use:
- Endpoint: `https://<account_id>.r2.cloudflarestorage.com`
- Or: `https://<account_id>.r2.cloudflarestorage.com` (from R2 dashboard)
- Credentials: `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`
- Bucket: `R2_BUCKET_NAME`
- Object key: `projects/{sessionId}.zip`

### Step 5: Trigger Export on Session Complete

**File:** `packages/server/src/ws/agent-handler.ts`

When `event.type === "session_complete"` and `finalStatus` is terminal (`completed`, `stopped`, `failed`):

1. Do not block the handler — run export asynchronously.
2. Call `exportSessionWorkspace(sessionId)` (fire-and-forget or background job).
3. On success: Project record created.
4. On failure: Log error; optionally retry or mark session with export_failed (future enhancement).

```typescript
// After updating session status and broadcasting:
if (["completed", "stopped", "failed"].includes(finalStatus)) {
  exportSessionWorkspace(sessionId).catch((err) =>
    console.error(`Export failed for session ${sessionId}:`, err)
  );
}
```

### Step 6: Download API

**File:** `packages/server/src/routes/sessions.ts`

Add route:

```
GET /api/sessions/:id/download
```

Logic:
1. Look up session and its project: `prisma.session.findUnique({ where: { id }, include: { project: true } })`
2. If no project, return 404 with message "Project not yet available" or "Export in progress"
3. Generate pre-signed URL (e.g. 15 min expiry) via R2/S3 `getSignedUrl`
4. Return 302 redirect to the pre-signed URL, or return JSON with `{ downloadUrl: "..." }` for client to open in new tab

Recommendation: Return JSON `{ downloadUrl }` — frontend can open in new tab or use `fetch` + blob for direct download with custom filename.

### Step 8: Environment Variables

**R2 setup:**
1. Cloudflare Dashboard → R2 → Create bucket (`coding-agent-projects`)
2. R2 → Manage API Tokens → Create token with Object Read & Write
3. Copy Account ID, Access Key ID, Secret Access Key

### Step 9: Frontend – Download Button

**File:** `packages/web/src/components/EventStream.tsx` (or session detail view)

- When session status is `completed` or `stopped`, show "Download Project" button
- On click: `GET /api/sessions/:id/download` → receive `downloadUrl`
- Open URL: `window.open(downloadUrl)` or trigger download with `document.createElement('a')`

Optional: Poll or subscribe to know when project is ready (export is async). Initial version can show button when session is terminal; if 404, show "Preparing download..." and retry after a few seconds.

### Step 10: Optional – Retention & Cleanup

Add a cron job or scheduled task to delete R2 objects and Project records where `expiresAt < now()`. For MVP, skip; projects persist indefinitely.

For production:
- Set `expiresAt = createdAt + 30 days` when creating Project
- Background job: list projects with `expiresAt < now`, delete from R2, delete Project record

## Edge Cases

| Case | Handling |
|------|----------|
| Empty workspace | Zip will contain 0–few files; still upload (small zip) |
| Export fails (R2 down, disk error) | Log; user sees "Download not available"; could add retry queue |
| Multi-turn sessions | Export only on terminal state (completed/stopped/failed), not waiting_for_user |
| Agent restart mid-session | Workspace in shared volume persists; session may fail but files remain until next session overwrites (per-session dirs prevent cross-session overwrite) |
| Very large project (e.g. 500MB) | Streaming zip + multipart upload; consider timeout tuning |
